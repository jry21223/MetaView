# Code Prompt Guidance

## Common

- 源代码是唯一真相来源，动画不得发明源码中不存在的行为。
- 控制流、变量变更、数据结构更新、调用返回都必须按执行顺序呈现。
- 高亮的代码行必须对应当前真实执行位置，不能只按讲解顺序跳转。
- 变量值只能在触发该变更的语句之后更新到画面上。
- 函数调用、递归展开、返回值传播必须有明确的进入和退出路径。
- 条件分支、循环继续、异常终止或 return 路径必须保持可追踪。
- 同名变量若处于不同作用域，必须区分其身份，不能混用一个视觉对象。
- 可视化不能暗示编译器优化、并行执行或隐式状态，除非源码明确表现。

## Planner

- 先拆成代码结构介绍、主执行路径、关键状态变化、终止或返回总结。
- 优先选用能覆盖核心分支和典型状态变化的输入，而不是只产生最终结果的输入。
- 对每个 scene 标注当前代码范围、当前调用层级和当前关注变量。
- 先找最容易失真的点，如循环迭代边界、递归回溯、可变对象共享、作用域切换。
- 若源码较长，只保留影响当前逻辑的行，省略部分必须不改变语义。
- 若存在多个函数协作，规划清晰的调用主线，不要在同一时刻并讲多条路径。
- 对分支类代码，至少覆盖一条进入路径和一条不进入路径的判定原因。
- 在分镜中提前决定何处展示栈帧、何处展示数据结构，避免信息层级混乱。

## Coder

- 代码区、状态区、数据结构区、调用栈区保持固定分区，减少观众视线跳跃。
- 代码高亮切换先于状态更新，让观众知道当前变化由哪一行触发。
- 变量变更采用局部替换或数值更新，保留对象身份，不频繁整体重绘。
- 数据结构变化要与对应语句同步，插入、删除、交换、覆盖分别可辨。
- 调用函数时显式压栈，返回时显式出栈，并同步展示返回值去向。
- 条件判断时先显示条件结果，再进入对应分支，不要直接跳到分支内部。
- 循环场景要让当前迭代变量、循环条件和退出时机始终可见。
- 多处同时更新时按真实先后播放，不把连续执行压缩成一帧结果。

## Critic

- 检查是否有任何动画行为无法在源码中找到对应语句。
- 检查代码高亮顺序是否与真实执行流一致。
- 检查变量值是否在错误时机更新，尤其是赋值前先变。
- 检查调用栈是否缺失入栈、返回或作用域区分。
- 检查循环和分支是否跳过判定条件，直接展示结果。
- 检查数据结构变化是否与语句一一对应，是否存在凭空变化。
- 检查关键代码行、变量名或数据结构是否被遮挡，导致无法核对。
- 检查终止路径、return 值和最终输出是否来源明确。

## Repair

- 若高亮顺序错了，只重排相关语句的高亮与更新顺序。
- 若变量提前变更，延后该更新到对应语句之后。
- 若缺失条件判定，补一个局部判断展示，不改主布局。
- 若调用栈不清，补一层栈帧或返回箭头，不重做整个执行流。
- 若数据结构变化过于突兀，改成局部插入、删除或替换动画。
- 若省略代码过多导致语义断裂，补最少必要的中间行。
- 若作用域混淆，给局部变量加作用域标签或单独栈帧。
- 若返回值来源不明，补从 callee 到 caller 的返回路径。
