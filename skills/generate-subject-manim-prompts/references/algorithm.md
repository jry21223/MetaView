# Algorithm Prompt Guidance

## Common

- 代码执行顺序高于视觉顺序，所有更新必须按真实控制流发生。
- 数组单元、指针、队列、栈、visited、DP 表都要保持对象身份稳定。
- 索引、边界、循环条件必须在变化前可见，变化后立即同步。
- 比较、交换、覆盖、入队出队、入栈出栈、返回都属于关键状态变化，不可只展示结果。
- 循环不变量和状态定义一旦出现，后续演示不得悄悄改变含义。
- 递归或回溯必须同时体现进入、展开、回退、返回，不能压扁成单步结论。
- 若算法依赖有序性、单调性、最优子结构或贪心条件，必须先立再用。
- 视觉上不能制造代码中不存在的并行、提前知道结果或跳过中间更新。

## Planner

- 先拆成初始化、主循环或主递归、关键更新、终止与返回几个阶段。
- 优先选择能暴露边界变化和状态转移的小输入，不选会掩盖步骤的大输入。
- 若有多种对象并行变化，先确定主视角，只让一个主因果链驱动画面推进。
- 对每个 scene 明确当前不变量、当前索引范围或当前待处理集合。
- 提前找出最易讲错的步骤，如 swap 前后、指针交错、visited 更新时机、DP 填表顺序。
- 遇到嵌套循环或递归树过大时，只保留代表性分支，但不能破坏真实执行顺序。
- 若 topic 涉及状态压缩、剪枝或惰性更新，先规划“何时显示隐含状态”。
- 在分镜里显式标记返回值来源，避免观众只看到最终答案。

## Coder

- 让核心数据结构固定在画面稳定区域，避免步骤推进时整体漂移。
- 指针、索引和高亮必须绑定到对应对象，移动与更新同步发生。
- 每次 compare、swap、overwrite 只驱动相关对象变化，不让无关对象抖动。
- 循环推进时先切换代码位置或逻辑焦点，再做数据更新，不要倒序播放。
- 队列、栈、visited、DP 表的增删改必须可局部追踪，不能整块重绘掩盖更新点。
- 递归调用用稳定的层级或栈表示，返回时沿原路径回收信息。
- 多对象同时变化时，用先后节奏表达因果，不把连续操作压成一次 Transform。
- 返回、终止、剪枝等控制流结束点要有明确的视觉停顿。

## Critic

- 检查是否有任何一步视觉顺序早于代码真实顺序。
- 检查索引、边界、visited、DP 状态是否在关键更新后保持一致。
- 检查是否跳过 compare、swap、overwrite、push/pop、return 等关键过渡。
- 检查递归、回溯或循环是否只展示结果而缺失展开和回退。
- 检查不变量、状态定义、最优值含义是否前后漂移。
- 检查多个对象是否因重绘或遮挡导致观众看不清当前更新位置。
- 检查动画是否暗示了源码里不存在的并行、预知或捷径。
- 检查终止条件和返回值是否有明确来源。

## Repair

- 若顺序错了，只重排相关过渡的先后，不改整段分镜。
- 若关键更新被跳过，补一个局部 scene 或中间帧显示该变化。
- 若指针、索引或边界不清，补绑定标签或局部高亮，不重做布局。
- 若数据结构被整块重绘，改成局部替换或局部移动以暴露更新点。
- 若递归或回溯被压缩，补入栈、回退或返回的最小可见过程。
- 若不变量未被维持，补一句状态说明或在对应阶段固定显示。
- 若返回值来源不明，补返回路径或结果汇总标记。
- 若遮挡了关键单元，只调整相关对象间距或分层。
